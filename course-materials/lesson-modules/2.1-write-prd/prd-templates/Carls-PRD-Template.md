# Carl 的 PRD 模板

# 问题对齐

## 问题和机会

用 1-2 句话描述我们尝试解决的问题。我应该能够单独阅读这个，并将价值/风险传达给其他人。

- 这对我们的客户和业务为什么很重要？
- 你有什么证据或见解来支持这个？
- 我们正在与哪些客户一起工作，或从哪些客户那里听到信息，来更好地理解这个问题？
- 为什么解决这个问题是紧迫的？为什么是现在？
- 为什么这很重要？

## **高级方向**

描述我们如何解决问题的大致形状。我应该眯着眼睛看，看到相同的形状。例如，如果问题是"发现新功能"，那么这可能是"一个相关功能的通知中心"。

我们考虑了哪些替代方案？为什么我们采用这个？

### **叙述**

**可选**：分享（假设的）故事来描绘客户今天的生活样子。描述常见和边缘用例，以便在设计解决方案时考虑。

## **目标**

1. *描述高级目标，最好按优先级排列，数量不要太多。*
2. *包括可衡量的（指标）和不可衡量的（感觉）目标*
3. *保持简洁*
4. *包括护栏指标*

## **非目标**

1. *列出我们不打算解决的明确区域*
2. *解释为什么它们不是目标*
3. *这些和目标一样重要和澄清*

# 解决方案对齐

## 关键特性

记录计划

1. *列出塑造解决方案的功能*
2. *最好按优先级排列*
3. *把它看作是绘制解决方案空间的周长*
4. *画边界，使团队可以专注于如何填充它*
5. *为特别大的项目的更多细节链接到子文档*
6. *质疑大小，看看较小的组件是否可以独立发货*

未来的考虑

1. *可选地列出你为以后保存的功能*
2. *这些可能会影响你现在如何构建*

### **关键流程**

显示客户的端到端体验。这可以是书面散文、流程图、屏幕截图或设计探索。这将因项目和团队而异。不要孤立地尝试做这个。与设计和工程一起完成。

这一部分随着时间的推移自然变得更具体。它可能从一些带注释的屏幕截图或故事开始。它可能变成具有验收标准的高度详细的需求。根据你的团队的运作方式进行调整。如果你有一个喜欢处理每个边缘情况的强势设计师，依靠他们。如果你有喜欢记录每个场景的详细工程师，深入了解验收标准。

这自然会随时间变化 - 这是可以的。当发生变化时，在[变更日志](https://docs.google.com/document/d/1mEMDcHmtQ6twzNlpvF-9maNlAcezpWDtCnyIqWkODZs/edit#heading=h.n2tmu89lqb7n)中记录并通知所有贡献者。

### **关键逻辑**

1. *列出指导设计和开发的规则*
2. *解决常见场景和边缘情况*
3. *通常比依靠设计来显示每个排列更容易写出这些*
4. *包括非功能性需求*
