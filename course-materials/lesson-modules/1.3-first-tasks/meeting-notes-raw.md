=== 会议 2:与 Mike(CTO)的利益相关者签到(上午 11:30) ===

mike(cto)想讨论通知系统

背景:
通知现在很混乱
- 太多的邮件(用户抱怨)
- 通知中的上下文不足
- 移动通知效果不佳
- 用户错过了重要的更新

Mike 的观点:
"工程团队在通知中淹没"
- 平均工程师每天收到 40-50 条通知
- 大多数是噪音
- 重要的东西被埋没了

讨论:

1. 智能批处理(将相似的通知分组)
   - mike: "就像 github 一样 - 多个事件的一封电子邮件"
   - 我:有道理,但需要定义什么被批处理

2. 通知首选项需要大检修
   - 当前:二进制(每个通知类型开/关)
   - 需要:细粒度(按项目、按人员、按紧急程度)

3. 摘要模式
   - 批处理非紧急通知
   - 每天发送一次(或一天两次?)
   - mike: "我想要早晨摘要,而不是不断的中断"

4. 时区感知通知
   - 不要在凌晨 2 点通知!
   - 尊重工作时间
   - mike 在他的上一家公司建造了这个

技术考虑:
- 通知服务当前是同步的(减慢 API)
- 需要移动到异步队列(redis?kafka?)
- mike 估计 2 个冲刺的工作
- 需要数据库模式变化(首选项表重新设计)

Mike 的担忧:
- 团队满负荷(移动应用工作)
- 这会推迟暗模式
- 建议 Q2 而不是 Q1?

我的看法:通知很重要,但暗模式更可见
sarah 可能不同意 - 需要讨论优先事项

行动:
- 我:起草通知改进的 PRD
- 我:与客户成功交谈 - 有多少票与通知相关?
- 我:竞争分析(其他工具如何处理这个?)
- mike:如果我们决定继续,技术规范

=== 会议 3:规划会议 - 模板库(下午 2 点) ===

头脑风暴会议模板库项目
与会者:我、jordan(设计)、alex(移动 PM)、jamie(工程主管)

背景:
用户需要永远才能开始
新用户流程:
1. 注册
2. 盯着空工作空间
3. ...现在怎么办?
4. 手动创建第一个项目
5. 手动创建第一个任务
6. 最后开始使用 SingTech(45 分钟后!)

目标:让用户更快地获得 SingTech 的价值

模板库概念:
用户可以使用的预构建项目模板
- "产品发布"
- "冲刺规划"
- "招聘渠道"
- "内容日历"
- "错误跟踪"
- 等等。

讨论:

jordan: "我们应该在注册后立即显示模板"
- 模态:"选择一个模板开始"
- 可以跳过(创建空白项目)
- 模板有真实的任务,不仅仅是结构

我: "我们需要多少个模板?"
- jordan: "从 5-7 开始,稍后扩展"
- 太多 = 选择的悖论
- 太少 = 没有足够的多样性

alex: "移动应用也应该有模板"
- 一致的体验
- 移动用户可能需要模板(屏幕小,难以设置)

jamie(工程): "模板数据结构?"
- json 文件?数据库记录?
- 需要版本控制(模板演进)
- 需要支持自定义字段(模板可能使用它们)

关注:

jamie: "模板很棘手"
- 用户自定义模板
- 模板更新不会影响现有项目
- 或者他们做吗?(可能是一个功能!)

我: "隐私/安全?"
- 企业客户可能想要私有模板
- 特定于团队的模板(不仅仅是全球)
- sarah 之前提到过这个

竞争研究注意:
- Asana 有模板库(巨大!100+ 个模板)
  - 按类别组织
  - 有些是"经 Asana 认证的"
  - 用户可以创建/共享模板
- Notion 有模板(内联、好 UX)
- ClickUp 有模板(令人困惑、太多选项)
- Linear:没有模板(有趣!他们只是有空白项目)

开放问题:
1. 模板发现 - 用户如何找到模板?
2. 模板自定义 - 创建项目前编辑?
3. 社区模板 - 让用户共享?
4. 模板分析 - 哪些模板受欢迎?

jamie 的技术估计:3-4 个冲刺
- 模板存储系统
- 模板呈现逻辑
- 用于浏览模板的 UI
- 从模板创建的 UI
- 版本控制系统(如果我们想要更新)

alex: "这能等到移动应用发布之后吗?"
- 团队能力问题再次出现
- 移动应用 = Q1 优先事项
- 这可能是 Q2
