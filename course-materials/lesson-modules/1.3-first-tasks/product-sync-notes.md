=== 通知讨论 ===

我展示通知重新设计提案

当前状态是坏的:
- 上个季度有 127 张关于通知的票
- 用户要么每天收到 50+ 条通知,要么关闭所有通知
- 42 种不同的通知类型(太多了!)
- 开/关偏好仅(不够细粒度)
- 同步送达 = 慢 API(200-400ms 延迟 - mike 说的)
- 企业客户在销售电话中抱怨

我的提案:
3 个等级 - 紧急/重要/fyi
- 紧急 = 立即(已分配、@提及、截止日期)
- 重要 = 摘要合格(评论、状态更改)
- fyi = 仅选择加入(赞、观察者、常规活动)

智能批处理 - 对相关通知进行分组
例如:同一任务上的 5 条评论 = 1 条通知而不是 5 条

摘要模式 - 每天或一天两次
但紧急仍然立即

时区感知 - 尊重工作时间(用户 tz 中的 9-6)
在工作时间外排队,稍后送达

更好的内容 - 显示任务标题 + 预览、谁做的、操作按钮

MIKE 的技术说明:
"需要异步队列,同步杀了我们"
- 迁移到 redis(我们已经使用它)
- 通知工作人员处理队列
- 3-4 周实施

需要的数据库更改:
"user_preferences 表有 42 个布尔列,不可扩展"
想要 jsonb 列
展示了示例模式(紧急/重要/fyi,有频道、摘要设置、安静时间)

迁移 = 2 周,不能翻转开关,需要逐步推出
将现有用户默认为"全部=重要"

"异步将改进性能,通知是最慢的 api 操作"
